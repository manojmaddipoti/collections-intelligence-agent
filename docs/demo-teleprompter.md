# YouTube Demo Teleprompter

Target length: 4 to 5 minutes.

## 0:00 - 0:25 Opening

On screen: Show the GitHub repository README and cover image.

Speech:

Hi, this is Collections Intelligence Agent, a multi-agent accounts receivable
assistant built for Kaggle's AI Agents Intensive Vibe Coding Capstone, in the
Agents for Business track.

The project helps finance teams analyze overdue customer exposure, review
accounts receivable aging, and draft collection notices while keeping a human
approval gate in the workflow.

All data shown in this demo is synthetic. No real customer, invoice, payment,
tax, or bank data is used anywhere in this repository.

## 0:25 - 1:10 Problem And Solution

On screen: Scroll through the README Problem and Solution sections.

Speech:

Finance teams often spend a lot of time reviewing aging reports, identifying
which accounts need attention, and writing follow-up messages. That work is
repetitive, but it is also sensitive.

If an assistant gives the wrong balance, exposes account details, or sends a
message before review, it can create real business risk.

This project addresses that by splitting the workflow across three agents. The
orchestrator routes requests, the financial analyst answers AR questions, and
the communications agent drafts collection notices. The communications agent
does not send anything. It only saves drafts for human review.

## 1:10 - 1:55 Architecture

On screen: Open `docs/media/architecture.svg`, then briefly show
`agents/agent.py`, `agents/financial_analyst/agent.py`, and
`agents/communications/agent.py`.

Speech:

The architecture is intentionally simple and reviewable.

The user interacts through ADK Web or the ADK CLI. The root orchestrator agent
decides whether a request belongs to financial analysis, communication
drafting, or draft approval.

The financial analyst and communications agents do not access the database
directly. Instead, both use a custom MCP server. That server owns the tool
layer, talks to SQLite, and enforces masking before any customer data reaches
agent context.

This separation is important. Sensitive handling is implemented in Python code,
not just as a prompt instruction.

## 1:55 - 2:20 Data And Safety

On screen: Show `scripts/seed_data.py`, then `mcp_server/server.py`.

Speech:

The demo database is generated by `scripts/seed_data.py`. It uses Faker with a
fixed seed and a fixed aging date, so the same synthetic customers, invoices,
and aging buckets are created every time.

In the MCP server, customer tax IDs and bank account numbers are masked
server-side. The helper function strips or redacts those fields before tool
results are returned to any agent.

The communications workflow also has a hard safety boundary. There is no
email-sending tool in this system. A draft starts as `pending_review`, and an
explicit human approval action can only change the internal status to
`approved`.

## 2:40 - 3:15 Financial Analysis Prompt

On screen: ADK Web.

Action prompt:

```text
Who are our top 3 most overdue accounts?
```

Speech:

For the first demo prompt, I ask: who are our top three most overdue accounts?

The orchestrator should route this to the financial analyst agent. The analyst
uses MCP tools to query overdue invoices and rank accounts by exposure.

In the response, I am looking for customer names, overdue amounts, invoice
counts, and aging context. The key point is that the model is not inventing the
finance data. It is using the MCP tool layer.

## 3:15 - 3:55 Draft Communication Prompt

On screen: ADK Web response, then enter the next prompt.

Action prompt:

```text
Draft a collection notice for the highest overdue account.
```

Speech:

Next, I ask the system to draft a collection notice for the highest overdue
account.

This is a compound workflow. The orchestrator first needs the financial context,
then it delegates the writing task to the communications agent.

The communications agent looks up the customer details and invoices, writes a
professional notice, and saves it using the draft communication tool.

Notice that the result is a draft. The system should make clear that this is
not sent to the customer.

## 3:55 - 4:25 Review And Approve Draft

On screen: ADK Web.

Action prompt:

```text
List draft communications.
```

Then approve the draft ID shown:

```text
Approve draft DRAFT-XXXXXXXX.
```

Speech:

Now I list draft communications. The draft appears with a `pending_review`
status.

To demonstrate the human-in-the-loop gate, I explicitly approve that draft by
ID.

Approval changes the internal database status to `approved`. It does not send
an email, enqueue a message, or hand the draft to another delivery system. This
is intentional: the project demonstrates safe drafting and review, not
automated outbound collections.
